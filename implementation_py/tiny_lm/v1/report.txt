0) TL;DR (fast onboarding)
- v1 studies latent sender->receiver communication (not English generation quality).
- Main claim is now robust under controlled OOD testing: message helps a lot when receiver lacks private info.
- Preferred protocol: fixed held-out visibility signatures across seed sweeps (`ood_signature_seed`) to reduce split noise.
- Strong baseline under that protocol (`seeds=42..51`, heldout `[2,4]`): `iid_comm_gain=0.3714 +/- 0.0359`, `ood_comm_gain=0.7672 +/- 0.0225`.

1) Version intent
- v1 is the latent-communication research branch.
- Main objective: learn communication protocols where agents exchange hidden vectors instead of English text.
- This branch is designed for controlled communication experiments, ablations, and analysis.

2) Research aims
- Demonstrate that a receiver can solve belief-state tasks better with a sender message than without one.
- Quantify communication utility with ablations.
- Track whether latent messages encode belief-relevant information.

3) Core hypothesis
- If sender has private information and receiver lacks it, a learned continuous message should increase task accuracy.
- Measured signal: `with_message_acc - zero_message_acc` (communication gain).

4) Current implementation status

Implemented foundations:
- `comm_data.py`
  - Synthetic false-belief episode generator.
  - Dataset for sender tokens, receiver tokens, and target belief location.
  - Explicit private info channel via visibility bits known to sender.

- `comm_model.py`
  - `SenderEncoder` produces continuous latent message vector.
  - `ReceiverPolicy` consumes receiver observations + latent message.
  - `CommAgents.forward(..., message_mode=...)` supports:
    - `with_message`
    - `zero_message`
    - `shuffled_message`

- `eval_comm.py`
  - Accuracy evaluation utilities for each message mode.
  - Combined metric dictionary including communication gain.

- `train_comm.py`
  - End-to-end training loop.
  - Periodic ablation-aware validation logging.
  - Final metrics report.

- `tests/`
  - `test_comm_data.py`: episode correctness and dataset shape contracts.
  - `test_comm_model.py`: forward shape, message mode behavior, and gradient flow.

5) Task design details

Episode structure:
- Object starts in an initial location.
- Object moves `n_moves` times.
- Protagonist may or may not see each move.

Belief update rule:
- If protagonist sees a move, belief updates to new location.
- If protagonist does not see a move, belief remains unchanged.

Observations:
- Receiver sees object location trajectory (initial + moved locations).
- Sender sees trajectory + visibility bits (private information).

Target:
- Final protagonist belief location (classification over `n_locations`).

6) Why this setup is useful for latent communication
- Receiver alone cannot fully recover the target because visibility is hidden.
- Sender has privileged state and must encode it compactly in the message.
- This creates a direct pressure to learn a useful latent protocol.

7) Metric plan

Primary metrics:
- `with_message_acc`

13) Terminology quick glossary (added 2026-04-21)
- In-distribution (IID): evaluation data sampled from the same pattern family used in training.
- Out-of-distribution (OOD): evaluation data sampled from a deliberately held-out pattern family.
- Visibility signature: a binary pattern over moves where `1` means protagonist saw that move and `0` means did not see it.
  - Example for `n_moves=3`: signature `010` means only the middle move was seen.
- Holdout ratio: fraction of feasible signatures reserved for OOD evaluation.
- OOD signature seed: seed used only for choosing held-out signatures; allows fixed OOD split while varying training seed.
- Leakage: when OOD-held-out signatures accidentally appear in train or IID validation data.
- Seed sweep: repeat full train/eval across multiple random seeds to measure robustness.
- Mean/std: aggregate center (`mean`) and variability (`std`) of a metric over seed runs.

14) OOD visibility-signature holdout implementation (2026-04-21)
- Added deterministic signature holdout selection and OOD dataset builder in `comm_data.py`.
  - New API: `build_datasets_with_signature_ood(config, holdout_ratio, ood_val_size, signature_seed)`.
  - New helper: `visibility_signature(saw_flags)`.
  - Guardrails:
    - `holdout_ratio` must be in `(0, 1)`.
    - At least two feasible signatures are required (so `protagonist_sees_prob` cannot be exactly `0` or `1` for this split strategy).
- Updated `train_comm.py` to train on in-distribution signatures and report both IID and OOD ablations.
  - New train config fields: `ood_val_size`, `ood_holdout_ratio`, `ood_signature_seed`.
  - Final metrics now include `iid_*` and `ood_*` prefixes plus held-out signature IDs.
- Added tests for deterministic selection and leakage prevention.
  - Command: `.venv/bin/python -m pytest -q v1/tests`
  - Result: `12 passed in 4.83s`.

15) Plain-language progress walkthrough (added 2026-04-21)

What we changed, in order, and why:
- Step 1: We confirmed the base communication setup works (sender message helps receiver).
  - Why: establish a working reference before changing evaluation protocol.
- Step 2: We added OOD evaluation via held-out visibility signatures.
  - Why: test generalization, not just same-distribution performance.
- Step 3: We ran a first multi-seed sweep and saw high OOD variance.
  - Why this mattered: some variance came from changing which signatures were held out between seeds.
- Step 4: We introduced `ood_signature_seed` to freeze held-out signatures across seed runs.
  - Why: isolate model/training randomness from split-selection randomness.
- Step 5: We reran 10 seeds with fixed heldout signatures and saw stable strong OOD communication gain.
  - Outcome: this is now the preferred robustness protocol.
- Step 6: We swept `message_dim` under the same fixed-heldout protocol.
  - Outcome: all tested dimensions (4/8/16/32) perform similarly well; `32` is slightly best in current runs.

How to read the current results quickly:
- `with_message_acc`: receiver performance with true sender message.
- `zero_message_acc`: receiver performance with message removed.
- `comm_gain`: how much message helps (`with_message_acc - zero_message_acc`).
- If `ood_comm_gain` stays clearly positive across many seeds with fixed heldout split, the latent communication claim is stronger.

Current practical conclusion:
- v1 now has a reproducible and defensible evaluation stack for latent communication.
- Main remaining unknown is not "does communication work" (it does), but "what exactly is encoded in the message" (next diagnostics/probe phase).
